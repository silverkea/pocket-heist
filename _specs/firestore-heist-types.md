# Spec for Firestore Heist Document Types

branch: fs-schemas
figma_component (if used): N/A

## User Story

As a developer building heist features, I want a shared set of types for Firestore heist documents so that data shapes are enforced consistently across all reads and writes at compile time.

## Trigger

A developer adds, reads, or updates a heist document in Firestore.

## Summary

Define the canonical TypeScript types for the Firestore `heists` collection. This gives every part of the codebase a single, authoritative contract for what a heist document looks like — covering the full document shape, the input shape for creating a new heist, and the partial shape for updating one. The types exist purely as a shared definition layer; they do not implement any data access logic themselves.

## Functional Requirements

- A `Heist` type defines the full shape of a heist document as it appears after being read from Firestore
- A heist document must include: a unique identifier, a title, a description, the creator's user ID, the creator's codename, a deadline (48 hours from creation), and a final status
- The `assignedTo` field (assignee user ID) and `assignedToCodename` field are optional — a heist may be created without an assignee and assigned later
- The `finalStatus` field is `null` when the heist is first created; it may later become either `"success"` or `"failure"` — no other values are permitted
- A `CreateHeistInput` type defines the shape passed when creating a new heist document — it excludes the document ID (generated by the database) and uses a server-side timestamp placeholder for `createdAt`
- An `UpdateHeistInput` type defines the shape passed when updating an existing heist — all fields are optional and `createdAt` is excluded (it must not change after creation)
- A Firestore converter for the `Heist` type handles the translation between raw Firestore data and the typed `Heist` shape — including converting Firestore `Timestamp` fields to JavaScript `Date` objects
- A `COLLECTIONS` constant exports the canonical Firestore collection name for heists
- All types and the converter are exported from a single barrel file

## Success Criteria

- A developer can import `Heist`, `CreateHeistInput`, `UpdateHeistInput`, and `heistConverter` from a single location
- TypeScript rejects a `CreateHeistInput` that is missing a required field such as `title` or `createdBy`
- TypeScript rejects a `finalStatus` value other than `null`, `"success"`, or `"failure"`
- A heist document read through the converter has `createdAt` and `deadline` as JavaScript `Date` objects, not raw Firestore timestamps
- A developer building a create-heist feature can omit `assignedTo` and `assignedToCodename` without a type error

## Out of Scope

- Data access functions (reading, writing, querying Firestore) — these belong in future features
- User document types — deferred (see Open Questions)
- Validation logic or runtime schema enforcement — types are compile-time only
- Business rules for setting or transitioning `finalStatus` — this belongs in a future feature

## Non-Functional Requirements

- Types must be co-located and importable from a single barrel export to avoid scattered imports across the codebase
- The converter must not throw or silently corrupt data when optional fields (`assignedTo`, `assignedToCodename`) are absent

## Security Considerations

- These are compile-time type definitions only — no data is read, written, or transmitted by this feature
- The types describe documents that contain user IDs (`createdBy`, `assignedTo`) — these are internal identifiers and are not considered PII under GDPR in isolation, but any feature that reads or displays them should apply appropriate access controls
- No secrets, credentials, or API keys are introduced by this feature

## Edge Cases & Constraints

- A heist with no assignee must not fail type-checking — `assignedTo` and `assignedToCodename` must both be optional
- The `finalStatus` union must be closed — no string other than `null`, `"success"`, or `"failure"` is valid
- The `deadline` field is a `Date` on the document type but will be stored as a Firestore `Timestamp` — the converter must translate both `createdAt` and `deadline` on read

## Acceptance Criteria

- Given a valid `CreateHeistInput` without `assignedTo`, TypeScript compiles without error
- Given a `CreateHeistInput` with `assignedTo` present, TypeScript also compiles without error
- Given a `finalStatus` value of `"pending"` or any string not in the union, TypeScript reports a type error
- Given a heist document snapshot, the converter returns a `Heist` with `createdAt` and `deadline` as `Date` instances
- Given a heist document snapshot with no `assignedTo` field, the converter returns a `Heist` without throwing
- Given an import from the barrel export, `Heist`, `CreateHeistInput`, `UpdateHeistInput`, `heistConverter`, and `COLLECTIONS` are all accessible

## Observability

Standard error handling applies. These are static type definitions and a data converter — there is no runtime logic to monitor beyond the converter's `fromFirestore` path, which should not throw on well-formed Firestore documents.

## Open Questions

None.

## Testing Guidelines

Types are compile-time constructs and are not directly unit-testable in the traditional sense. Testing should focus on the runtime converter:

- The converter returns a `Heist` with `createdAt` as a `Date`
- The converter returns a `Heist` with `deadline` as a `Date`
- The converter returns a `Heist` without throwing when `assignedTo` and `assignedToCodename` are absent from the snapshot
- The converter maps `id` from the snapshot's document ID, not a field in the document body
